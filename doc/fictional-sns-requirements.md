# 架空SNSアプリ 要件定義 

## 目的

ユーザーが投稿すると、多数のBOT/AIユーザーからいいね・コメント・通知が届き、人気インフルエンサーのような反応を体験できるモバイルSNSアプリを作る。

React Native（Expo）で開発する。**対象プラットフォームは iOS のみ**（Android は対象外）。

## 重要な前提

- X/Twitterをそのまま複製するのは避ける。
- 慣れたSNS操作、タイムライン、投稿、通知、プロフィールの体験は参考にする。
- アプリ名、ロゴ、色、アイコン、文言、細部UIはオリジナルにする。
- 実在人物になりすますBOTは作らない。
- BOTであることはアプリ内で管理上区別できるようにする。

## MVPの機能

### 1. アカウント作成

- メールアドレスまたはユーザー名で登録。
- 表示名、ユーザーID、プロフィール画像、自己紹介を設定。
- 初回登録後、チュートリアルで「あなたの投稿には多くの反応が届きます」という体験を自然に案内する。

### 2. ログイン

- メールアドレス/ユーザーIDとパスワードでログイン。
- MVPでは簡易認証でも可。
- 本番ではFirebase Authを使う。

### 3. 投稿

- テキスト投稿。
- 画像付き投稿。
- 投稿文字数は280文字から始める。
- タイムラインに自分の投稿とBOT投稿を表示。

### 4. 画像投稿

- カメラロールから画像を選択。
- 1投稿あたり最大4枚まで。
- MVPでは画像圧縮とアップロードを必須にする。

### 5. タイムライン

- 自分の投稿、BOTの投稿、BOTからの反応を表示。
- 新着順。
- 投稿カードには以下を表示する。
  - アイコン
  - 表示名
  - ユーザーID
  - 投稿日時
  - 本文
  - 画像
  - コメント数
  - いいね数
  - 共有/保存相当のアクション

### 6. コメント

- 投稿詳細画面でコメント一覧を表示。
- ユーザーはBOT投稿にもコメントできる。
- ユーザーがコメントすると、BOTが返信することがある。
- BOTコメントは複数トーンを用意する。
  - 称賛
  - 共感
  - 質問
  - 軽いツッコミ
  - 深掘り

### 7. いいね

- 投稿とコメントにいいねできる。
- 自分の投稿にはBOTから自動的にいいねが増える。
- いいね数は即時に少し増え、その後数分かけて段階的に増える。

### 8. 通知

通知はこのアプリの中核機能として作り込む。

通知タブで表示する通知:

- BOTがあなたの投稿にいいねした
- BOTがあなたの投稿にコメントした
- BOTがあなたをフォローした
- BOTがあなたのコメントに返信した
- 投稿が一定数のいいねに到達した
- 投稿が急に伸びている
- 人気BOTがあなたの投稿に反応した

通知の演出:

- 投稿直後に最初の反応を早く出す。
- その後、波のように通知を増やす。
- 重要通知は通常通知より目立たせる。
- 未読バッジを表示する。
- プッシュ通知は後続フェーズで実装する。

## BOT/AI仕様

### BOTの役割

- ユーザー以外の世界を構成する。
- タイムラインに投稿する。
- ユーザーの投稿にいいね、コメント、フォロー、返信をする。
- 人気インフルエンサー感を演出する。

### BOTタイプ

- 一般ユーザーBOT
- 熱心なファンBOT
- 業界人風BOT
- ミーム系BOT
- クリエイター風BOT
- 人気インフルエンサーBOT

### BOTプロフィール生成

BOTにはランダムでアイコン、表示名、ユーザーID、自己紹介、口調を付与する。

名前は「SNSに自然にいそうな架空名」にする。実在人物、実在ブランド、著名人、既存インフルエンサーに見える名前は避ける。

表示名の例:

- Riku
- Mio
- Haru
- Sora
- Nagi
- Yuna
- Ren
- Aoi
- Kaito
- Lina
- Emma
- Kai
- Nao
- Rei
- Yuki
- Mika

ユーザーIDの例:

- riku_notes
- mio_daily
- haru_log
- sora_vibes
- nagi_room
- yuna_clip
- ren_creates
- aoi_days
- kai_memo
- lina_loop

アイコン生成ルール:

- MVPではローカルまたはStorage上のプリセット画像からランダム選択する。
- アイコンカテゴリは人物風イラスト、抽象アイコン、風景、ペット以外の小物、カラーパターンなどに分ける。
- 実在人物の写真に見える素材は避ける。
- 同じBOTが毎回違うアイコンにならないよう、bot_idに紐づけて固定する。
- 後続フェーズでAI生成アイコンに差し替えられるようにする。

BOTプロフィール項目:

- display_name
- username
- avatar_url
- bio
- bot_persona
- tone
- popularity_rank
- created_at

### BOT反応ロジック

ユーザー投稿後、以下のような段階で反応する。

1. 0-10秒: 数件のいいね、1-2件の短いコメント
2. 10-60秒: いいねがまとまって増える、コメントも増える
3. 1-5分: 人気BOTから反応、通知が強くなる
4. 5分以降: 断続的な追加反応

### AIコメント生成

MVPでは以下のどちらかを選ぶ。

- テンプレート生成: 安い、速い、オフラインでも開発しやすい
- LLM API生成: 自然だがコストと安全対策が必要

最初はテンプレート生成で作り、後からLLM APIに差し替えられる設計にする。

## デザイン方針

X/Twitter風の情報設計:

- 縦スクロールのタイムライン
- 投稿カード中心
- 返信・再投稿相当・いいね・共有のアクション列
- 通知タブ重視
- プロフィール画面とフォロー関係

オリジナル化する要素:

- アプリ名
- ブランドカラー
- ロゴ
- アイコンセット
- 投稿アクション名
- 通知文言
- BOTの世界観

推奨デザイン:

- 白背景/黒背景のライト・ダーク両対応
- ブランドカラーは青系そのままではなく、シアン、ライム、コーラルなどを組み合わせる
- 角丸や余白はモバイルSNSとして自然な範囲にする
- 通知タブだけは反応の熱量が伝わるように強弱をつける

## 技術方針

### フロントエンド

- React Native（**iOS のみ**）
- Expo
- TypeScript
- React Navigation
- Zustand または Redux Toolkit
- TanStack Query
- Expo Image Picker
- Expo Notifications
- 実機／シミュレータ確認は iOS を前提とする

### バックエンド

Firebaseを採用する。

- Firebase Auth
  - アカウント作成
  - ログイン
  - ユーザー識別
- Cloud Firestore
  - 投稿
  - コメント
  - いいね
  - 通知
  - BOTイベント
- Firebase Storage
  - 投稿画像
  - プロフィール画像
  - BOTアイコン
- Cloud Functions for Firebase
  - BOT反応イベント生成
  - 通知作成
  - カウント更新
  - 不正データ補正
- Firebase Cloud Messaging
  - 後続フェーズのプッシュ通知
- Firebase Analytics
  - 投稿率、通知開封率、継続率の分析

MVPではFirestoreとStorageを中心に実装し、BOTイベントの一部はアプリ内モックから始めてもよい。MVP後半でCloud Functionsへ移す。

## リスクと対策

### 依存性・メンタル面

承認欲求を刺激する設計は強い体験になる一方、依存性が高くなる。

対策:

- 通知頻度の設定を用意する。
- BOT反応であることを規約や設定内に明示する。
- 深夜通知を抑制できるようにする。

### なりすまし

対策:

- 実在人物名や実在ブランドに見えるBOTを避ける。
- BOTプロフィールは架空で生成する。

### ストア審査

対策:

- ユーザー生成コンテンツ対策を用意する。
- 通報/ブロック機能を後続フェーズではなくMVP後半に前倒しする。
- BOT/AI体験の説明を明確にする。
